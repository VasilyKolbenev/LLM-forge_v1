# Claude Code Execution Prompt

## Назначение

Этот файл нужен как прямой операционный prompt для Claude Code. Его задача не обсуждать общие идеи, а выполнять работу по проекту поэтапно, с явными критериями готовности и без хаотичных рефакторингов.

Использовать вместе с файлом `docs/CLAUDE_CODE_ROADMAP.md`.

## Роль Claude Code

Ты работаешь как senior engineer и delivery-агент внутри этого репозитория.

Твоя задача:

- изучать текущую реализацию перед изменениями;
- находить минимальный надежный путь для выполнения фазы;
- вносить изменения непосредственно в код;
- запускать релевантные проверки;
- фиксировать результат, риски и следующие шаги.

Нельзя ограничиваться только анализом или общими советами, если фазу можно реально продвинуть кодом.

## Главная цель

Последовательно довести проект от `demo-ready / dev-ready` состояния до более устойчивого `production-capable` состояния без разрушения текущей рабочей базы.

## Главные правила работы

1. Работай строго фазами по `docs/CLAUDE_CODE_ROADMAP.md`.
2. Не смешивай несколько больших фаз в одном заходе без острой необходимости.
3. Перед кодом сначала прочитай файлы-ориентиры текущей фазы.
4. Делай минимально достаточные изменения, но доводи их до рабочего состояния.
5. После каждого смыслового блока запускай релевантные тесты, сборку или smoke-check.
6. Если встречаешь риск миграции, совместимости или безопасности, сначала уменьши blast radius, потом расширяй изменение.
7. Не делай косметический рефакторинг вне рамок текущей фазы, если он не нужен для завершения задачи.
8. Если обнаруживается новый существенный долг, документируй его в `docs/CLAUDE_CODE_ROADMAP.md` или отдельном рабочем md-файле.

## Непереговорные инженерные требования

- Сначала надежность и корректность, потом удобство и косметика.
- Persistence, state handling и security важнее UI-полировки.
- Любой новый persistent path должен иметь понятную схему, migration story и тесты.
- Любой новый runtime flow должен иметь предсказуемый terminal state.
- Любая auth/key логика не должна светить секреты в логах, URL или UI без крайней необходимости.
- Любое крупное изменение должно сопровождаться короткой записью: что сделано, что проверено, какие риски остались.

## Как брать работу

### Шаг 1. Определи текущую фазу

Сначала выбери одну фазу из `docs/CLAUDE_CODE_ROADMAP.md`.

Если нет явного указания от пользователя, используй такой приоритет:

1. Фаза 1. Persistence layer.
2. Фаза 2. Durable jobs and sessions.
3. Фаза 3. Security hardening.
4. Фаза 4. Unified pipeline execution.
5. Фаза 5. Remote compute hardening.
6. Фаза 6. Frontend decomposition and tests.
7. Фаза 7. Protocols and assistant productionization.
8. Фаза 8. Test harness and CI stabilization.

### Шаг 2. Сначала изучи контекст

Перед правками обязательно:

- прочитай файлы-ориентиры фазы;
- найди смежные тесты;
- проверь, нет ли уже похожего механизма в проекте;
- оцени, можно ли сделать изменение как вертикальный срез, а не как большой незавершенный рефакторинг.

### Шаг 3. Сформируй маленький исполнимый план

План должен быть коротким и прикладным, например:

1. Ввести новый storage adapter.
2. Перевести один store на новый backend.
3. Добавить migration path.
4. Обновить тесты.
5. Прогнать проверки.

### Шаг 4. Реализуй, а не только опиши

Если пользователь не попросил только анализ, нужно реально менять код, запускать команды и доводить задачу до результата.

### Шаг 5. Проверь результат

После реализации обязательно укажи:

- какие файлы изменены;
- какие проверки запущены;
- что прошло, а что не удалось проверить;
- какие риски или хвосты остались.

## Что считать хорошим результатом

Хороший результат по каждой итерации выглядит так:

- есть реальный кодовый прогресс в рамках выбранной фазы;
- изменения не разлетаются по всему проекту без необходимости;
- есть тесты или хотя бы узкий smoke-check;
- следующая итерация понятна и логично продолжает предыдущую.

Плохой результат:

- длинный анализ без изменений, когда изменения были возможны;
- смешивание persistence, security, UI и remote compute в одном большом diff;
- большой рефакторинг без migration path;
- выводы без проверки кодом или тестами.

## Фокус по фазам

### Фаза 1. Persistence layer

Фокус:

- убрать критичные `load -> mutate -> save` path;
- ввести SQLite и migration bootstrap;
- сначала перевести `ExperimentStore` и `PromptStore`;
- потом расширять на workflows, tracking, auth и compute config.

Definition of Done:

- данные читаются и пишутся через единый persistence layer;
- есть миграция из старого JSON формата;
- есть тесты на запись, чтение, обновление и конкурентные сценарии.

### Фаза 2. Durable jobs and sessions

Фокус:

- убрать критичную зависимость от in-memory state;
- сделать restart-safe jobs, runs и sessions;
- добавить reconcile stale state.

Definition of Done:

- после рестарта состояние либо восстанавливается, либо корректно переводится в terminal state;
- зависшие jobs/runs можно reconcile-ить предсказуемо.

### Фаза 3. Security hardening

Фокус:

- убрать ключи из URL;
- перестать логировать plaintext secrets;
- выровнять auth policy;
- покрыть это тестами.

Definition of Done:

- query-string key bootstrap удален;
- в логах нет секретов;
- security regression tests проходят.

### Фаза 4. Unified pipeline execution

Фокус:

- свести sync и websocket execution к одному engine path.

Definition of Done:

- нет отдельной дублирующей step-loop логики в route layer;
- статусы и ошибки одинаковы в обоих режимах.

### Фаза 5. Remote compute hardening

Фокус:

- command safety;
- remote lifecycle;
- exit-code capture;
- heartbeat;
- artifact metadata.

Definition of Done:

- remote runs надежно завершаются и оставляют наблюдаемое состояние;
- ошибки различимы и диагностируемы.

### Фаза 6. Frontend decomposition and tests

Фокус:

- разрезать крупные экраны;
- снизить bundle pressure;
- добавить frontend test stack.

Definition of Done:

- большие страницы разбиты на более мелкие части;
- есть frontend tests для ключевых сценариев;
- bundle warning уменьшен или обоснован.

### Фаза 7. Protocols and assistant productionization

Фокус:

- убрать demo-only statefulness и hardcoded behavior там, где это уже мешает продукту.

Definition of Done:

- protocols имеют управляемый lifecycle;
- assistant и site chat меньше зависят от хардкода.

### Фаза 8. Test harness and CI stabilization

Фокус:

- стабилизировать тесты, особенно на Windows;
- сделать CI понятным и воспроизводимым.

Definition of Done:

- локальные и CI проверки дают чистый, повторяемый сигнал о состоянии проекта.

## Формат ответа после каждой итерации

Используй короткий, но предметный формат:

### Что сделано

- список реальных изменений;
- без воды и без пересказа всего diff.

### Что проверено

- какие тесты, build или smoke-check запускались;
- что прошло;
- что не удалось прогнать.

### Риски и хвосты

- что осталось незавершенным;
- какие ограничения есть у текущей итерации.

### Следующий лучший шаг

- один конкретный следующий шаг в рамках текущей фазы.

## Рекомендуемый старт прямо сейчас

Начни с Фазы 1 и выполни первый законченный вертикальный срез:

1. Спроектируй минимальный SQLite persistence layer.
2. Добавь bootstrap/schema initialization.
3. Добавь migration path из существующих JSON-файлов.
4. Переведи `ExperimentStore` на новый backend.
5. Переведи `PromptStore` на новый backend.
6. Обнови или добавь тесты.
7. Запусти релевантные проверки.
8. Коротко зафиксируй результат и оставшиеся риски.

Скопируй и используй такой старт:

```md
Работай по документам `docs/CLAUDE_CODE_ROADMAP.md` и `docs/CLAUDE_CODE_EXECUTION_PROMPT.md`.

Возьми в работу Фазу 1 из roadmap. Сначала изучи текущие `ExperimentStore`, `PromptStore`, связанный tracking и существующие тесты. Затем реализуй минимальный, но рабочий SQLite persistence layer с migration path из JSON, переведи на него сначала `ExperimentStore` и `PromptStore`, обнови тесты и прогони релевантные проверки.

Не ограничивайся анализом: внеси реальные изменения в код. После завершения дай короткий отчет в формате:
- что сделано;
- что проверено;
- какие риски остались;
- какой следующий лучший шаг.
```
